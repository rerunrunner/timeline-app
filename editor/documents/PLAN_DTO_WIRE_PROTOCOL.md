# DTO and Wire Protocol Plan

## Current State Analysis

### Problem Statement

The Timeline Data Editor API currently exposes JPA entities directly:

**Current Flow:**
```
Frontend → Controller → Entity (direct) → Jackson → JSON → Frontend
Frontend → JSON → Jackson → Entity (direct) → Repository → Database
```

**Issues:**
- ❌ **Tight Coupling**: API contract bound to database schema
- ❌ **Security Risks**: Internal fields exposed (database IDs, internal state)
- ❌ **Circular References**: Jackson struggles with bidirectional relationships
- ❌ **Lazy Loading Issues**: Serialization triggers unintended queries
- ❌ **Inflexible API**: Can't evolve API without changing database
- ❌ **No Versioning**: Breaking database changes break API
- ❌ **Poor Abstraction**: Business logic mixed with persistence concerns

### Example Problem: Reveal Entity

```java
@Entity
public class Reveal {
    private Long id;
    private Long eventId;  // Exposes internal ID
    
    @ManyToOne(fetch = FetchType.LAZY)
    @JsonIgnore  // Hides relationship from API
    private Event event;
    
    @Column(name = "displayed_date")
    private String displayedDate;
    
    // When serialized: only IDs exposed, no nested objects
    // When deserialized: must manually resolve IDs to entities
}
```

**Current API Response:**
```json
{
  "id": 166,
  "eventId": 42,  // Just an ID, no event details
  "episodeId": 14,
  "displayedDate": "December 13, 2023",
  "createdAt": "2024-03-15T10:30:00",
  "updatedAt": "2024-03-15T11:45:00"
}
```

**Problems:**
1. Frontend must make separate requests to get event/episode details
2. Can't embed event description without custom serializer
3. Changing entity structure breaks API
4. Can't have different representations for list vs. detail views

## Target State

### DTO-Based Architecture

**Target Flow:**
```
Frontend → Controller → DTO → Mapper → Entity → Repository → Database
Database → Repository → Entity → Mapper → DTO → Controller → Frontend
```

**Benefits:**
- ✅ **Decoupling**: API contract independent of database schema
- ✅ **Flexibility**: Different DTOs for create/update/response
- ✅ **Security**: Control exactly what's exposed
- ✅ **Versioning**: Old DTOs work even after schema changes
- ✅ **Optimization**: Embed or flatten data as needed
- ✅ **Clarity**: Clear separation of concerns

### Example: Reveal DTOs

**RevealDTO (Response):**
```java
public class RevealDTO {
    private Long id;
    
    // Embedded event summary (not just ID)
    private EventSummaryDTO event;
    
    // Embedded episode info
    private EpisodeSummaryDTO episode;
    
    private String displayedDate;
    private String displayedTitle;
    private String displayedDescription;
    private String screenshotFilename;
```

**API Response:**
```json

  "event": {
    "id": 42,
    "shortDescription": "Halmeoni gets dinner",
    "timelineShortId": "TL1"
  },
  "episode": {
    "id": 14,
    "number": 14,

  },

  "displayedTitle": "Halmeoni gets dinner",

  "absolutePlayTime": 22579

```

**Benefits:**
- Frontend gets all needed data in one request
- API response optimized for UI needs
- Database schema can change independently
- Can add computed fields without touching entity

## DTO Strategy

### DTO Types

#### 1. Response DTOs
**Purpose**: Serialize data for API responses  
**Direction**: Entity → DTO → JSON

```java
@Schema(description = "Reveal response")
public class RevealDTO {
    // All fields for complete representation
    // Read-only fields included (id, timestamps)
    // Nested objects for convenience
    // Computed fields
}
```

#### 2. Create DTOs
**Purpose**: Deserialize creation requests  
**Direction**: JSON → DTO → Entity

```java
@Schema(description = "Create reveal request")
public class RevealCreateDTO {
    // Only fields needed for creation
    // No ID (generated by database)
    // No timestamps (set by system)
    // Validation annotations
    
    @NotNull
    private Long eventId;
    
    @NotNull
    @Min(0)
    private Integer episodeTime;
}
```

#### 3. Update DTOs
**Purpose**: Deserialize update requests  
**Direction**: JSON → DTO → Entity update

```java
@Schema(description = "Update reveal request")
public class RevealUpdateDTO {
    // Only updatable fields
    // All fields optional (partial update)
    // Validation annotations
    
    private Long eventId;
    private Integer episodeTime;
    private String displayedDate;
}
```

#### 4. Summary DTOs
**Purpose**: Lightweight representation for embedding  
**Direction**: Entity → DTO (minimal)

```java
@Schema(description = "Event summary")
public class EventSummaryDTO {
    // Minimal fields for reference

    private String shortDescription;
    private String timelineShortId;
}
```

#### 5. List DTOs
**Purpose**: Optimized for list views  
**Direction**: Entity → DTO (streamlined)


## DTO Mapping Strategy

### Option 1: Manual Mapping (Recommended for Start)

**Pros:**
- ✅ Full control over mapping logic
- ✅ Easy to debug
- ✅ No magic/reflection
- ✅ Type-safe

**Cons:**
- ❌ Boilerplate code
- ❌ Manual maintenance

**Implementation:**

```java
@Component
public class RevealMapper {
    
    @Autowired
    private EventMapper eventMapper;
    
    @Autowired
    private EpisodeMapper episodeMapper;
    
    // Entity → DTO
    public RevealDTO toDTO(Reveal entity) {
        if (entity == null) return null;
        
        RevealDTO dto = new RevealDTO();
        dto.setId(entity.getId());
        dto.setEvent(eventMapper.toSummaryDTO(entity.getEvent()));
        dto.setEpisode(episodeMapper.toSummaryDTO(entity.getEpisode()));
        dto.setDisplayedDate(entity.getDisplayedDate());
        dto.setDisplayedTitle(entity.getDisplayedTitle());
        dto.setDisplayedDescription(entity.getDisplayedDescription());
        dto.setScreenshotFilename(entity.getScreenshotFilename());
        
        // Computed fields
        if (entity.getScreenshotFilename() != null) {
            dto.setScreenshotUrl("/api/v1/reveals/" + entity.getId() + 
                                "/image/" + entity.getScreenshotFilename());
        }
        
        return dto;
    }
    
    // CreateDTO → Entity
    public Reveal toEntity(RevealCreateDTO dto, Event event, Episode episode) {
        Reveal entity = new Reveal();
        entity.setEvent(event);
        entity.setEpisode(episode);
        entity.setEpisodeTime(dto.getEpisodeTime());
        entity.setDisplayedDate(dto.getDisplayedDate());
        entity.setDisplayedTitle(dto.getDisplayedTitle());
        entity.setDisplayedDescription(dto.getDisplayedDescription());
        return entity;
    }
    
    // UpdateDTO → Entity (merge)
    public void updateEntity(RevealUpdateDTO dto, Reveal entity) {
        if (dto.getEpisodeTime() != null) {
            entity.setEpisodeTime(dto.getEpisodeTime());
        }
        if (dto.getDisplayedDate() != null) {
            entity.setDisplayedDate(dto.getDisplayedDate());
        }
        if (dto.getDisplayedTitle() != null) {
            entity.setDisplayedTitle(dto.getDisplayedTitle());
        }
        if (dto.getDisplayedDescription() != null) {
            entity.setDisplayedDescription(dto.getDisplayedDescription());
        }
        // Note: Relationship updates handled separately
    }
    
    // List mapping
    public List<RevealDTO> toDTOList(List<Reveal> entities) {
        return entities.stream()
            .map(this::toDTO)
            .collect(Collectors.toList());
    }
}
```

### Option 2: MapStruct (Future Enhancement)

**Pros:**
- ✅ Generates mapping code at compile time
- ✅ Reduces boilerplate
- ✅ Type-safe
- ✅ Performance (no reflection)

**Cons:**
- ❌ Learning curve
- ❌ Complex custom mappings can be tricky
- ❌ Build-time dependency

**Example:**

```java
@Mapper(componentModel = "spring", uses = {EventMapper.class, EpisodeMapper.class})
public interface RevealMapper {
    
    @Mapping(target = "event", source = "event")
    @Mapping(target = "episode", source = "episode")
    @Mapping(target = "screenshotUrl", expression = "java(buildScreenshotUrl(entity))")
    RevealDTO toDTO(Reveal entity);
    
    @Mapping(target = "id", ignore = true)
    @Mapping(target = "event", source = "event")
    @Mapping(target = "episode", source = "episode")
    Reveal toEntity(RevealCreateDTO dto, Event event, Episode episode);
    
    default String buildScreenshotUrl(Reveal entity) {
        if (entity.getScreenshotFilename() == null) return null;
        return "/api/v1/reveals/" + entity.getId() + "/image/" + entity.getScreenshotFilename();
    }
}
```

**Recommendation**: Start with manual mapping (Option 1), migrate to MapStruct (Option 2) later if boilerplate becomes burden.

```java
        // Partial update (only non-null fields)

        return entities.stream()
            .map(this::toDTO)
            .collect(Collectors.toList());
    }
}
```

#### 3.3 Handle Circular Dependencies

**Problem**: Event → Reveal → Event (circular)

**Solution**: Use summary DTOs to break cycle

```java
        EventSummaryDTO dto = new EventSummaryDTO();

        dto.setShortDescription(entity.getShortDescription());
```

### Phase 4: Service Layer Integration (Week 3-4)

#### 4.1 Update Services to Use DTOs

**Pattern:**

**Before:**
```java
@Service
public class RevealService {
    public Reveal createReveal(Reveal reveal) {
        return revealRepository.save(reveal);
    }
    
    public List<Reveal> findAll() {
        return revealRepository.findAll();
    }
}
```

**After:**
```java
@Service
public class RevealService {
    
    @Autowired
    private RevealRepository revealRepository;
    
    @Autowired
    private RevealMapper revealMapper;
    
    @Autowired
    private EventRepository eventRepository;
    
    @Autowired
    private EpisodeRepository episodeRepository;
    
    @Transactional
    public RevealDTO create(RevealCreateDTO createDTO) {
        // Resolve relationships
        Event event = eventRepository.findById(createDTO.getEventId())
            .orElseThrow(() -> new NotFoundException("Event not found"));
        Episode episode = episodeRepository.findById(createDTO.getEpisodeId())
            .orElseThrow(() -> new NotFoundException("Episode not found"));
        
        // Map DTO → Entity
        Reveal entity = revealMapper.toEntity(createDTO);
        entity.setEvent(event);
        entity.setEpisode(episode);
        
        // Save and return DTO
        Reveal saved = revealRepository.save(entity);
        return revealMapper.toDTO(saved);
    }
    
    @Transactional(readOnly = true)
    public List<RevealDTO> findAll(String search) {
        List<Reveal> entities = search != null 
            ? revealRepository.findBySearchTerm(search)
            : revealRepository.findAll();
        
        return revealMapper.toDTOList(entities);
    }
    
    @Transactional(readOnly = true)
    public RevealDTO findById(Long id) {
        Reveal entity = revealRepository.findById(id)
            .orElseThrow(() -> new NotFoundException("Reveal not found"));
        return revealMapper.toDTO(entity);
    }
    
    @Transactional
    public RevealDTO update(Long id, RevealUpdateDTO updateDTO) {
        Reveal entity = revealRepository.findById(id)
            .orElseThrow(() -> new NotFoundException("Reveal not found"));
        
        // Update relationships if provided
        if (updateDTO.getEventId() != null) {
            Event event = eventRepository.getReferenceById(updateDTO.getEventId());
            entity.setEvent(event);
        }
        if (updateDTO.getEpisodeId() != null) {
            Episode episode = episodeRepository.getReferenceById(updateDTO.getEpisodeId());
            entity.setEpisode(episode);
        }
        
        // Update scalar fields
        revealMapper.updateEntity(updateDTO, entity);
        
        // Save and return DTO
        Reveal saved = revealRepository.save(entity);
        return revealMapper.toDTO(saved);
    }
    
    @Transactional
    public void delete(Long id) {
        if (!revealRepository.existsById(id)) {
            throw new NotFoundException("Reveal not found");
        }
        revealRepository.deleteById(id);
    }
}
```

#### 4.2 Service Layer Patterns

**Pattern 1: DTO In, DTO Out**
```java
public RevealDTO create(RevealCreateDTO dto) { ... }
```
- **Pros**: Clean API, decoupled from entities
- **Cons**: Service can't be reused for internal logic

**Pattern 2: Entity In, DTO Out**
```java
public RevealDTO create(Reveal entity) { ... }
```
- **Pros**: Reusable service methods
- **Cons**: Entities leak into service layer

**Pattern 3: Hybrid**
```java
public RevealDTO create(RevealCreateDTO dto) { ... }  // Public API
private Reveal createInternal(Reveal entity) { ... }  // Internal use
```
- **Pros**: Best of both worlds
- **Cons**: More methods

**Recommendation**: Start with Pattern 1 (DTO In, DTO Out), add Pattern 3 if internal reuse needed.

#### Deliverables:
- All services updated to use DTOs
- Service layer tests updated
- Business logic separated from presentation

```java
        return revealService.findById(id);
    }
    
    @Operation(summary = "Update reveal")
    @PutMapping("/{id}")
    public RevealDTO updateReveal(
        @PathVariable Long id,
        @Valid @RequestBody RevealUpdateDTO updateDTO
    ) {
        return revealService.update(id, updateDTO);
    }

        revealService.delete(id);
    }
}
```

#### 5.2 Controller Layer Responsibilities

Controllers should **only**:
- Accept DTOs from HTTP requests
- Validate DTOs (`@Valid`)
- Call service layer
- Return DTOs as HTTP responses
- Handle HTTP-specific concerns (status codes, headers)

Controllers should **never**:
- Access repositories directly
- Contain business logic
- Map entities to DTOs (that's service layer)
- Handle transactions

#### Deliverables:
- All controllers updated to use DTOs
- Controllers thin (delegating to services)
- HTTP concerns properly handled

```java
@Schema(description = "Reveal response")
public class RevealDTO {
    // ... other fields ...
    
    @Schema(description = "Absolute playtime (computed)")
    private Integer absolutePlayTime;
}

// In mapper
public RevealDTO toDTO(Reveal entity) {
    RevealDTO dto = new RevealDTO();
    // ... map fields ...
    
    // Compute absolute playtime
    int absoluteTime = 0;
    for (int i = 1; i < entity.getEpisode().getNumber(); i++) {
        Episode prevEpisode = episodeRepository.findByNumber(i);
        if (prevEpisode != null) {
            absoluteTime += prevEpisode.getDuration();
        }
    }
    absoluteTime += entity.getEpisodeTime();
    dto.setAbsolutePlayTime(absoluteTime);
    
    return dto;
}
```

## Migration Strategy

### Backward Compatibility

**Option 1: Parallel Endpoints**
- Keep old endpoints unchanged (`/api/tables/reveal`)
- Add new DTO-based endpoints (`/api/v1/reveals`)
- Deprecate old endpoints after frontend migration
- Remove old endpoints after grace period

**Option 2: Adapter Layer**
- Add compatibility layer that converts entities to DTOs
- Old endpoints return DTOs (breaking change)
- Frontend updated in single release
- Remove adapter after verification

**Recommendation**: Option 1 (parallel endpoints) for safer migration.

### Rollout Plan

**Week 1-2: Internal Refactoring**
- Create DTOs
- Create mappers
- Update services (internal only)
- Test thoroughly

**Week 3: Expose New API**
- Create new controllers with `/api/v1` prefix
- Keep old controllers active
- Update Swagger docs

**Week 4: Frontend Migration**
- Update frontend to use new endpoints
- Use generated TypeScript client
- Test thoroughly

**Week 5: Deprecation**
- Mark old endpoints as deprecated
- Add warnings in logs
- Monitor usage

**Week 6: Cleanup**
- Remove old endpoints
- Remove entity serialization annotations
- Final testing
